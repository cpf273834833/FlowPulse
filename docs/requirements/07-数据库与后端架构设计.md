# FlowPulse 数据库与后端架构设计

## 一、后端质量目标

后端按企业级应用标准设计，当前单进程部署，代码边界按未来可拆微服务设计。

基本要求：

1. 不使用内存数据替代数据库。
2. 不使用万能 JSON 表替代业务表。
3. Controller 不直接写业务逻辑。
4. 业务模块不能直接越过 API 契约访问其它模块的 Mapper。
5. DTO、领域模型、持久化实体边界清晰。
6. 异常统一处理，错误码统一管理。
7. 租户 ID、主键 ID 均使用 32 位字符串。
8. 时间字段统一使用毫秒时间戳。
9. 密码、私钥等敏感字段不明文回传前端；平台 API Key 与 OMP API Key 按当前产品规则允许明文保存和回显。
10. 代码结构支持后续按模块拆为微服务并水平扩容。

## 二、当前 Maven Module 规划

```text
flow-pulse-server
├── flow-pulse-common
├── flow-pulse-security
├── flow-pulse-tenant
├── flow-pulse-management-api
├── flow-pulse-management-service
├── flow-pulse-resource-api
├── flow-pulse-resource-service
├── flow-pulse-infrastructure-api
├── flow-pulse-infrastructure-service
├── flow-pulse-executor-api
├── flow-pulse-executor-service
├── flow-pulse-integration-api
├── flow-pulse-integration-service
└── flow-pulse-web
```

模块职责：

| 模块 | 职责 |
|---|---|
| flow-pulse-common | 通用响应、异常、错误码、ID、分页、基础工具 |
| flow-pulse-security | 登录用户、菜单权限、按钮权限、平台权限适配 |
| flow-pulse-tenant | 租户上下文、租户隔离策略 |
| flow-pulse-management-api | 环境、区域、平台配置等管理域接口契约 |
| flow-pulse-management-service | 环境、区域、平台配置领域规则与持久化 |
| flow-pulse-resource-api | 统一资源中心接口契约 |
| flow-pulse-resource-service | Topic、Index、Spark 作业、应用、主机等资源身份管理 |
| flow-pulse-infrastructure-api | Kafka、Spark、Elasticsearch 基础设施接口契约 |
| flow-pulse-infrastructure-service | 基础设施连接配置、连接测试、资源同步入口 |
| flow-pulse-executor-api | 执行节点接口契约 |
| flow-pulse-executor-service | SSH/Agent 执行能力、OMP 同步节点、连接测试 |
| flow-pulse-integration-api | 外部系统访问接口契约 |
| flow-pulse-integration-service | OMP、平台、Kafka、ES、Spark、SSH 等客户端实现 |
| flow-pulse-web | HTTP Controller、鉴权入口、异常转换、部署包入口 |

依赖方向：

```text
flow-pulse-web
  -> 业务 service module
  -> 业务 api module
  -> common / security / tenant
```

业务模块之间需要协作时，优先依赖对方的 `api` 模块，不直接访问对方数据库 Mapper。

## 三、业务模块内部结构

以 `flow-pulse-management-service` 为例：

```text
com.flowpulse.management
├── application
│   └── EnvironmentRegionService.java
├── domain
│   └── model
│       ├── EnvironmentEntity.java
│       ├── RegionEntity.java
│       └── PlatformConfigEntity.java
└── infrastructure
    └── persistence
        └── mapper
            ├── EnvironmentMapper.java
            ├── RegionMapper.java
            └── PlatformConfigMapper.java
```

分层职责：

| 分层 | 职责 |
|---|---|
| api/request | 前端请求 DTO |
| api/response | 前端响应 DTO |
| application | 用例编排、事务边界、跨实体业务校验 |
| domain/model | 当前领域内的业务对象 |
| infrastructure/persistence/mapper | MyBatis Mapper，属于持久化适配层 |
| resources/mapper | MyBatis XML SQL |
| web/controller | HTTP 接口入口，只调用 application service |

Mapper 放在 `infrastructure.persistence.mapper`，原因是 MyBatis Mapper 是数据库访问实现，不属于领域模型，也不属于应用用例本身。

## 四、环境与区域当前落地位置

接口契约：

```text
flow-pulse-management-api
└── src/main/java/com/flowpulse/management/api
    ├── request
    └── response
```

业务实现：

```text
flow-pulse-management-service
└── src/main/java/com/flowpulse/management
    ├── application
    ├── domain/model
    └── infrastructure/persistence/mapper
```

HTTP 入口：

```text
flow-pulse-web
└── src/main/java/com/flowpulse/web/controller/EnvironmentRegionController.java
```

数据库脚本：

```text
flow-pulse-web
└── src/main/resources/db/schema-h2.sql
```

已实现规则：

1. 环境编码租户内唯一。
2. 区域编码在同一环境内唯一。
3. 区域分为管理区和计算区。
4. 计算区必须归属于同一环境下的管理区。
5. 删除环境前必须先删除区域。
6. 删除管理区前必须先删除下属计算区。
7. 平台与 OMP 配置可以绑定任意区域。
8. 每个区域最多一组平台与 OMP 配置。
9. 平台 API Key、OMP API Key 明文保存并明文回显。

## 五、数据库选型

一期使用 H2 文件模式验证产品能力，表结构按真实业务表设计，后续兼容迁移到 MySQL 或 PostgreSQL。

```yaml
spring:
  datasource:
    driver-class-name: org.h2.Driver
    url: jdbc:h2:file:./data/flowpulse;MODE=MySQL;DATABASE_TO_LOWER=TRUE;AUTO_SERVER=TRUE
    username: flowpulse
    password: flowpulse
```

## 七、水平扩容优先级

后续优先独立扩容：

| 能力 | 原因 |
|---|---|
| metric-service | 指标写入、查询、历史保留压力大 |
| job-service | 资源同步、脚本执行、指标采集任务耗时且并发高 |
| alert-service | 阈值计算、状态流转、故障聚合频繁 |
| notification-service | 第三方通知平台不稳定，需要隔离重试 |
| integration-service | 外部系统调用慢，容易拖垮主流程 |

环境区域、基础设施、执行节点属于管理类低频能力，当前可以在单体中部署，但仍按模块边界实现。

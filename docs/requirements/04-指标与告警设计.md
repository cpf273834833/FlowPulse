# FlowPulse 指标与告警设计

## 一、指标设计原则

1. 指标值统一为浮点数。
2. 状态类指标也保存为浮点数，通过展示映射字典表达状态含义。
3. 指标定义与采集配置分离。
4. 指标采集配置与拓扑展示配置分离。
5. 节点指标、连线指标、基础设施指标、应用指标都需要统一纳入指标中心维护。

## 二、指标类型

| 指标类型 | 说明 |
|---|---|
| 基础设施指标 | Kafka、Spark、ES 实例自身状态 |
| 资源节点指标 | Topic、Index、Spark 作业、应用节点等 |
| 真实关系指标 | 两个资源之间的数据流向关系指标 |
| 画布展示指标 | 某个拓扑上选择展示的指标 |
| 衍生指标 | 由源节点指标、目标节点指标、连线指标组合计算得到 |

## 三、采集方式

一期支持多种采集方式：

| 采集方式 | 一期要求 |
|---|---|
| 服务端直接采集 | 支持 |
| SSH 到指定机器采集 | 支持 |
| Agent 下发采集 | 预留，一期不做完整实现 |
| 第三方接口采集 | 可作为采集器扩展 |

自定义脚本：

1. 支持 Shell。
2. 支持 Python。
3. 脚本输出需要能解析成浮点数。
4. 失败、超时、无效输出需要有明确状态。

脚本在哪里执行，由指标采集配置指定：

1. 选择执行节点。
2. 选择执行方式：SSH 或后续 Agent。
3. 指定超时时间。
4. 指定环境变量、参数、工作目录等。

## 四、内置指标

内置指标可以通过内置代码逻辑或内置脚本实现。

设计原则：

1. 对 Kafka、ES 等有成熟客户端的场景，优先使用官方客户端机制。
2. 对 Spark REST API 这类 HTTP 可查询场景，使用稳定 API。
3. 内置指标必须登记为指标定义，不允许散落在代码中不可发现。
4. 内置指标和自定义指标在阈值、展示、告警上使用同一套模型。

## 五、连接测试能力分项

基础设施连接测试需要展示分项能力：

1. 基础连接是否可用。
2. 认证是否通过。
3. 资源发现是否可用。
4. 指标采集是否可用。

原因：有些账号可能只有资源查询权限，没有指标查询权限；页面需要让用户知道是哪一类能力不可用。

## 六、Kafka Group 规则设计

### 6.1 背景

很多 Spark 作业消费 Kafka 时，Group ID 不是固定值：

1. 每次作业重启生成新的 Group ID，前缀相同，后缀可能是时间戳。
2. 每个 Executor 使用不同 Group ID，但前缀相同。
3. 一个 Spark 作业可能需要监控多个 Group ID。
4. 作业重启后旧 Group ID 和新 Group ID 命名规范相同。

### 6.2 一期设计

消费组选择器用于确定哪些物理 Kafka Group 属于当前真实关系。

支持：

1. 固定 Group ID。
2. 前缀匹配。
3. 正则匹配。
4. 自定义表达式限定范围。

自定义表达式需要支持时间规则，例如：

1. 正则捕获 Group ID 中的时间部分。
2. 定义时间解析格式。
3. 以 Spark 作业启动时间为基准。
4. 只选取捕获时间大于等于 Spark 作业启动时间的 Group ID。

这样正则匹配到的 Group ID 才有明确范围，不会误把历史 Group 算入当前链路。

## 七、指标衍生能力

一期需要支持指标衍生。

场景：

应用主节点消费 Kafka Topic 时，链路是否正常可能不是单个指标能直接说明，需要组合：

1. Topic 是否有消息。
2. 主节点是否在线。
3. 主节点是否为当前主节点。
4. Consumer Group 是否活跃。
5. Lag 是否增长。
6. 最后提交 offset 时间。

因此需要支持：

1. 节点指标 + 目标节点指标 + 连线指标组合成新的连线指标。
2. 衍生指标用于页面直观观察和故障判定。
3. 衍生指标保存定义、表达式、输入指标和输出指标。
4. 衍生指标输出仍然是浮点数。

## 八、阈值配置

### 8.1 阈值归属

建议阈值按“资源/真实关系 + 指标”配置，不按拓扑画布配置。

理由：

1. 同一个真实资源可能出现在多个拓扑中。
2. 阈值代表监控判断规则，不是画布样式。
3. 这样告警结果在不同拓扑中保持一致。

### 8.2 拓扑内配置阈值

拓扑页面必须支持阈值配置入口，因为用户只有看着拓扑才最容易配置。

规则：

1. 拓扑内配置阈值时，先查询该阈值是否被其他拓扑引用。
2. 如果没有被使用，可以直接配置。
3. 如果已被使用，需要提示在哪些拓扑中被使用。
4. 用户确认后再修改共享阈值。
5. 仍需要提供统一阈值管理页面。

### 8.3 告警级别

保留六级：

| 级别 | 说明 |
|---|---|
| 正常 | 无异常 |
| 提醒 | 轻微风险 |
| 警告 | 明确异常趋势 |
| 错误 | 已影响链路 |
| 紧急 | 严重影响或中断 |
| 未知 | 数据缺失、状态不可判断 |

## 九、告警通知

通知规则：

1. 初始状态是正常。
2. 只要告警级别发生变更就发送通知。
3. 告警升级发送通知。
4. 告警降级发送通知。
5. 告警恢复发送通知。
6. 通知渠道一期为第三方通知平台接口。
7. 后续可扩展告警触发动作。

## 十、故障事件

故障事件用于把一次异常过程中的多次级别变化关联为一次故障。

聚合键建议包含：

1. tenantId。
2. scopeType：RESOURCE 或 RELATION。
3. scopeId。
4. metricCode。
5. thresholdId。

生命周期：

1. 正常 -> 非正常：创建故障事件。
2. 非正常 -> 非正常：更新故障事件，追加历史。
3. 非正常 -> 正常：关闭故障事件，记录恢复。
4. 正常 -> 正常：不产生事件。

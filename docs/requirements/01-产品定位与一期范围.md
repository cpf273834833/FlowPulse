# FlowPulse 产品定位与一期范围

## 一、产品定位

FlowPulse 是数据流可观测平台，核心不是单点监控，而是围绕“数据从哪里来、经过哪里处理、写到哪里去”建立可观测的数据流拓扑。

平台需要让用户通过一张完整的数据流拓扑图回答下面几个问题：

1. 当前数据流是否正常。
2. 数据在哪一个节点或哪一段关系上出现异常。
3. 异常具体是什么指标触发的。
4. 异常是资源自身问题、链路关系问题，还是上游中断导致当前环节无数据。
5. 告警从什么时候开始、级别如何变化、什么时候恢复。

## 二、一期目标

一期目标是完成可部署、可使用、可演进的基础版本，优先保证核心领域边界正确，而不是堆叠临时功能。

一期必须具备：

1. 环境与区域管理。
2. 基础设施管理，支持 Kafka、Spark、Elasticsearch。
3. 执行节点管理，支持人工新增和从区域 OMP 同步。
4. 资源维护与同步，包括 Kafka Topic、ES Index、Spark 作业等。
5. 数据流列表与数据流拓扑。
6. 人工创建拓扑，节点从已有资源选择。
7. 真实关系维护，拓扑画布与真实关系解耦。
8. 指标定义与指标采集配置。
9. Shell/Python 自定义指标。
10. 阈值管理与拓扑内阈值配置入口。
11. 告警状态、告警历史、故障生命周期。
12. 第三方通知平台对接预留并支持一期通知。

## 三、一期明确边界

### 3.1 一期需要做

| 能力 | 一期要求 |
|---|---|
| 拓扑创建 | 完全人工创建 |
| 节点来源 | 只能从已有资源选择，不允许建图时顺便创建资源 |
| 资源维护 | 独立页面维护 |
| 数据流边界 | 允许跨区域，不允许跨环境 |
| 告警级别 | 正常、提醒、警告、错误、紧急、未知 |
| 指标历史 | 默认保留 7 天，页面支持配置 |
| 通知 | 调第三方通知平台接口 |
| 关键节点 | 一期不区分关键与非关键 |
| Agent | 预留，不做完整 Agent 采集 |
| 权限 | 先支持菜单和按钮级权限控制 |

### 3.2 一期不做

| 能力 | 说明 |
|---|---|
| 自动发现完整拓扑 | 一期人工建图 |
| 自动根因推断闭环 | 一期先展示状态、指标、关联告警和解释 |
| 自定义应用角色 | 一期应用实例角色先支持主节点、从节点 |
| 完整 Agent 体系 | 后续迭代 |
| 机器学习异常检测 | 后续迭代 |
| 告警自动动作编排 | 一期只做通知，动作扩展预留 |

## 四、命名

产品名称可继续使用 FlowPulse。

理由：

1. 名称已经能表达“数据流脉搏”和“实时状态感知”。
2. 历史项目已有一定识别度。
3. 新项目可以延续名称，但产品定义和技术实现需要重新设计。

## 五、部署约束

1. 前后端仍基于公司脚手架。
2. 前后端代码放在 `D:\codex` 下。
3. 构建出的 tar 包、全量包、补丁包需要保持当前公司 OMP 部署方式。
4. 后端一期可使用 H2 看效果，但架构上必须支持后续替换 MySQL/PostgreSQL 等生产数据库。
5. 后端使用 MyBatis，不能使用临时内存数据或万能 JSON 存储替代业务表。
